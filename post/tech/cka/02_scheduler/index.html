Taints, Tolerations

  • taints
    • set on the nodes
  • toleartions
    • set on the pods
  • taints and toleration does not tell the pod to go to a particular node
  • node to only accept pod with certain tolerations

Taints

설정

  • kubectl taint nodes node-name key=value:taint-effect
  • taint-effect :
    • NoSchedule
      • not to be scheduled, if they do not tolerate the taint
    • PreferNoSchedule
      • system will be try to avoid placing a pod on the node, not guaranteed
    • NoExecute
      • new pods will not be scheduled on the node
      • existing pods on the node will be evicted if the do not tolertae the taint

확인

  • describe & grep
    1k describe node node01 |grep -i taints
2Taints:             <none>

제거

  • kubectl taint nodes node-name key=value:taint-effect-
    1kubectl taint nodes master/controlplane node-role.kubernetes.io/master:NoSchedule-

Node Affinity

  • nodeSelector vs affinity
    • two are working equally, schedule pods to Large node.

설정

 1apiVersion:
 2kind:
 3metadata:
 4  name:
 5spec:
 6  containers:
 7  - name:
 8    image:
 9  affinity:
10    nodeAffinity:
11      requireDuringSchedulingIgnoredDuringExecution:
12        nodeSelectorTerms:
13        - matchExpressions:
14          - key:
15            operator:
16            values:
  • operator
    • In
      • guarantee to schedule in values
    • NotIn
      • guarantee not to schedule in values
    • Exists
      • given key exists or not

Node Affinity Types

  • available
    • requireDuringSchedulingIgnoredDuringExecution
    • preferredDuringSchedulingIgnoredDuringExecution
  • planned
    • requireDuringSchedulingRequiredDuringExecution
  • two states in the life cycle of pod
    • DuringScheduling
      • Required
        • must be
      • Preferred
        • best to
    • DuringExecution
      • Ignored
        • if they are scheduled, will not impact them.
      • Required
        • pod will be evicted

Static PODs

Static PODs vs DaemonSets

  • static pod
    • created directly by kubelet

Config

  1. --pod-manifeset-path=/etc/Kubernetes/manifest
  2. --config=kubeconfig.yaml
    1# kubeconfig.yaml
2staticPodPath: /etc/Kubernetes/manifest

파일 위치

First idenity the kubelet config file:

1root@controlplane:~# ps -aux | grep /usr/bin/kubelet
2root      3668  0.0  1.5 1933476 63076 ?       Ssl  Mar13  16:18 /usr/bin/kubelet --bootstrap-kubeconfig=/etc/kubernetes/bootstrap-kubelet.conf --kubeconfig=/etc/kubernetes/kubelet.conf --config=/var/lib/kubelet/config.yaml --network-plugin=cni --pod-infra-container-image=k8s.gcr.io/pause:3.2
3root      4879  0.0  0.0  11468  1040 pts/0    S+   00:06   0:00 grep --color=auto /usr/bin/kubelet
4root@controlplane:~#

From the output we can see that the kubelet config file used is /var/lib/kubelet/config.yaml

Next, lookup the value assigned for staticPodPath:

1root@controlplane:~# grep -i staticpod /var/lib/kubelet/config.yaml
2staticPodPath: /etc/kubernetes/manifests
3root@controlplane:~#

As you can see, the path configured is the /etc/kubernetes/manifests directory.

삭제

First, let's identify the node in which the pod called static-greenbox is created. To do this, run:

1root@controlplane:~# kubectlget pods --all-namespaces -o wide  | grep static-greenbox
2defaultstatic-greenbox-node01                 1/1Running   0          19s     10.244.1.2   node01       <none>           <none>
3root@controlplane:~#

From the result of this command, we can see that the pod is running on node01.

Next, SSH to node01 and identify the path configured for static pods in this node.

Important: The path need not be /etc/kubernetes/manifests.

Make sure to check the path configured in the kubelet configuration file.

1root@controlplane:~# ssh node01
2root@node01:~# ps -ef |  grep /usr/bin/kubelet
3root       752   654  0 00:30 pts/0    00:00:00 grep --color=auto /usr/bin/kubelet
4root     28567     1  0 00:22 ?        00:00:11 /usr/bin/kubelet --bootstrap-kubeconfig=/etc/kubernetes/bootstrap-kubelet.conf --kubeconfig=/etc/kubernetes/kubelet.conf --config=/var/lib/kubelet/config.yaml --network-plugin=cni --pod-infra-container-image=k8s.gcr.io/pause:3.2
5root@node01:~# grep -i staticpod /var/lib/kubelet/config.yaml
6staticPodPath: /etc/just-to-mess-with-you
7root@node01:~#

Here the staticPodPath is /etc/just-to-mess-with-you

Navigate to this directory and delete the YAML file:

1root@node01:/etc/just-to-mess-with-you# ls
2greenbox.yaml
3root@node01:/etc/just-to-mess-with-you# rm -rf greenbox.yaml
4root@node01:/etc/just-to-mess-with-you#

Exit out of node01 using CTRL + D or type exit. You should return to the controlplane node. Check if the static-greenbox pod has been deleted:

1root@controlplane:~# kubectl get pods --all-namespaces -o wide  | grep static-greenbox
2root@controlplane:~#
